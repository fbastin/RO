\section{Logiciels d'optimisation}

\subsection{IOR Tutorial}

Le logiciel IOR Tutorial, qui accompagne le livre {\sl Introduction to Operational Research} \citep{HillLieb01} est disponible gratuitement à l'adresse \url{http://highered.mcgraw-hill.com/sites/0073017795/student_view0/ior_tutorial.html}.
Le logiciel est fourni sous la forme d'un exécutable sous l'environnement Microsoft Windows, et comme archive Java pour les autres plate-formes.
Il est par conséquent utilisable sur la majorité des systèmes actuels.
Le logiciel est intéressant pour rapidement tester des programmes d'optimisation simples, mais ne peut être considéré comme un solveur à part entière.
Son interface graphique le rend néanmoins adapté pour une expérimentation rapide avec les exemples introduits dans lecours.

\subsection{Julia et JuMP}

Julia est un langage de programmation moderne, performant et de haut niveau. JuMP est un paquet de modélisation algébrique pour Julia, permettant de formuler et résoudre des problèmes d'optimisation mathématique.
En pratique, l'utilisation de Julia et JuMP permet de :
\begin{itemize}
\item
spécifier la structure du modèle d'optimisation ;
\item
spécifier et calculer les données qui entrent dans le modèle ;
\item
résoudre ce modèle en appelant différents solveurs (HiGHS, Ipopt, Gurobi, etc.) ;
\item
analyser et afficher les résultats de la solution ;
\item
conduire une analyse de sensibilité ou comparative.
\end{itemize}
Julia et JuMP sont entièrement gratuits et open-source. Ils peuvent être installés sur la majorité des systèmes d'exploitation actuels (Windows, macOS, Linux).

\subsubsection{Formulation d'un problème simple}

\begin{example}
\label{jump_simple}
Considérons le problème d'optimisation linéaire
\begin{align*}
\max_x \ & 109 x_1 + 90 x_2 + 115 x_3 \\
\st & x_1 + x_2 + x_3 \leq 100; \\
& 6x_1 + 4x_2 + 8x_3 \leq 500; \\
& x_1,\ x_2,\ x_3 \geq 0.
\end{align*}
Le script Julia décrivant ce problème est constitué des parties suivantes :
\begin{enumerate}
\item
chargement des paquets requis (\verb|using JuMP, HiGHS|) ;
\item
définition du modèle et choix du solveur ;
\item
déclaration des variables de décision et de leurs bornes ;
\item
déclaration de la fonction objectif ;
\item
déclaration des contraintes ;
\item
lancement de la résolution (\verb|optimize!|) et affichage de la solution.
\end{enumerate}
Ce problème peut être formulé en Julia avec JuMP comme suit :
\begin{verbatim}
using JuMP
using HiGHS

# Initialisation du modèle
model = Model(HiGHS.Optimizer)

# Déclaration des variables (positives)
@variable(model, x1 >= 0)
@variable(model, x2 >= 0)
@variable(model, x3 >= 0)

# Déclaration de la fonction objectif (maximisation)
@objective(model, Max, 109*x1 + 90*x2 + 115*x3)

# Déclaration des contraintes
@constraint(model, Equation2, x1 + x2 + x3 <= 100)
@constraint(model, Equation3, 6*x1 + 4*x2 + 8*x3 <= 500)

# Résolution
optimize!(model)
\end{verbatim}
Détaillons les instructions.
\end{example}

\subsubsection{Spécification des variables}

JuMP utilise la macro \verb|@variable| pour déclarer les variables d'un modèle.
Dans l'exemple précédent, nous avons déclaré des variables non-négatives (positives) de la manière suivante :
\begin{verbatim}
@variable(model, x1 >= 0)
@variable(model, x2 >= 0)
@variable(model, x3 >= 0)
\end{verbatim}
Contrairement à GAMS, il n'est pas nécessaire d'introduire une variable explicite pour représenter l'objectif (comme \verb|Z| dans l'ancien modèle) : l'objectif est défini directement comme une expression mathématique sous la macro \verb|@objective|.

JuMP permet de restreindre facilement le domaine des variables lors de leur déclaration. Voici quelques exemples :\\
\begin{tabular}{ll}
\verb|@variable(model, x)| & variable libre (non restreinte) \\
\verb|@variable(model, x >= 0)| & variable non négative \\
\verb|@variable(model, x <= 0)| & variable non positive \\
\verb|@variable(model, x, Bin)| & variable binaire (dans $\lbrace 0, 1\rbrace$) \\
\verb|@variable(model, x, Int)| & variable entière \\
\verb|@variable(model, l <= x <= u)| & variable bornée entre $l$ et $u$
\end{tabular}

\subsubsection{Dénomination des contraintes}

En JuMP, les contraintes sont spécifiées à l'aide de la macro \verb|@constraint|. Contrairement à GAMS, il n'y a pas besoin de déclarer préalablement les noms des contraintes dans une section séparée : on peut les nommer directement au moment de leur écriture. 
Dans notre exemple :
\begin{verbatim}
@constraint(model, Equation2, x1 + x2 + x3 <= 100)
@constraint(model, Equation3, 6*x1 + 4*x2 + 8*x3 <= 500)
\end{verbatim}
Ici, \verb|Equation2| et \verb|Equation3| sont les noms donnés aux contraintes, ce qui permet d'y faire référence ultérieurement (par exemple pour obtenir leur valeur duale). Notez qu'il est également possible d'écrire des contraintes anonymes en omettant le nom.

Les opérateurs de comparaison en JuMP utilisent la syntaxe mathématique standard de Julia :\\
\begin{tabular}{ll}
\verb|==| & contrainte d'égalité \\
\verb|<=| & inférieur ou égal \\
\verb|>=| & supérieur ou égal
\end{tabular}

\subsubsection{Spécification du modèle et du solveur}

En JuMP, un modèle est un objet Julia créé à l'aide de la fonction \verb|Model()|. C'est lors de cette création que l'on indique au modèle quel solveur d'optimisation utiliser. Par exemple :
\begin{verbatim}
# Utilisation du solveur linéaire et mixte HiGHS
model = Model(HiGHS.Optimizer)
\end{verbatim}
Le choix du solveur dépend de la nature du problème :
\begin{itemize}
\item
Pour des problèmes linéaires (\verb|LP|) ou linéaires en nombres entiers (\verb|MIP|), le solveur open-source \verb|HiGHS| est une excellente option par défaut.
\item
Pour des problèmes non-linéaires (\verb|NLP|), on se tournera généralement vers des solveurs spécialisés comme \verb|Ipopt| :
\begin{verbatim}
using Ipopt
model = Model(Ipopt.Optimizer)
\end{verbatim}
\end{itemize}
La fonction objectif est ensuite déclarée à l'aide de la macro \verb|@objective|, en indiquant le sens de l'optimisation (\verb|Max| ou \verb|Min|) :
\begin{verbatim}
@objective(model, Max, 109*x1 + 90*x2 + 115*x3)
\end{verbatim}
Une fois le modèle complètement défini (variables, contraintes, objectif), la résolution est lancée par l'appel à la fonction :
\begin{verbatim}
optimize!(model)
\end{verbatim}

\subsubsection{Accès à la solution et affichage}

Après l'appel à \verb|optimize!(model)|, nous pouvons vérifier le statut de résolution et interroger le modèle pour récupérer les valeurs optimales :
\begin{itemize}
\item
Le statut de terminaison du solveur s'obtient avec \verb|termination_status(model)|. Il indique si le problème a été résolu à l'optimalité (\verb|OPTIMAL|), s'il est irréalisable (\verb|INFEASIBLE|), ou non borné (\verb|DUAL_INFEASIBLE|).
\item
La valeur optimale de la fonction objectif s'obtient avec \verb|objective_value(model)|.
\item
La valeur optimale d'une variable s'obtient avec \verb|value(variable)|.
\item
Le coût réduit d'une variable s'obtient avec \verb|reduced_cost(variable)|.
\item
La valeur duale d'une contrainte (le multiplicateur de Lagrange) s'obtient avec \verb|shadow_price(contrainte)| ou \verb|dual(contrainte)|.
\end{itemize}

Nous pouvons écrire un script d'affichage pour présenter proprement les résultats :
\begin{verbatim}
println("Statut de résolution : ", termination_status(model))
println("Valeur de l'objectif (Z) = ", objective_value(model))
println("Variables de décision :")
println("  x1 = ", value(x1), " (coût réduit = ", reduced_cost(x1), ")")
println("  x2 = ", value(x2), " (coût réduit = ", reduced_cost(x2), ")")
println("  x3 = ", value(x3), " (coût réduit = ", reduced_cost(x3), ")")
println("Variables duales (contraintes) :")
println("  Equation2 = ", shadow_price(Equation2))
println("  Equation3 = ", shadow_price(Equation3))
\end{verbatim}
Ce qui produit à l'écran :
\begin{verbatim}
Statut de résolution : OPTIMAL
Valeur de l'objectif (Z) = 9950.0
Variables de décision :
  x1 = 50.0 (coût réduit = -0.0)
  x2 = 50.0 (coût réduit = -0.0)
  x3 = 0.0 (coût réduit = -13.0)
Variables duales (contraintes) :
  Equation2 = 52.0
  Equation3 = 9.5
\end{verbatim}
La valeur duale de 52,0 associée à \verb|Equation2| signifie que si la disponibilité de la contrainte 2 (la ressource de terre) augmentait d'une unité (de 100 à 101), le profit total augmenterait de 52,0. Le coût réduit de -13,0 associé à $x_3$ (le coton) indique que si nous étions forcés de cultiver une quantité infinitésimale de coton, le profit global diminuerait de 13,0 par unité produite.

\subsubsection{Sommations}

En Julia, la somme $\sum_j x_j$ se traduit à l'aide de la fonction \verb|sum| et d'un générateur :
\begin{verbatim}
sum(x[j] for j in J)
\end{verbatim}
Pour des sommes multiples ou imbriquées comme $\sum_j \sum_i x_{ji}$ (ou $\sum_{j,i} x_{ji}$), on peut écrire :
\begin{verbatim}
sum(x[j, i] for j in J for i in I)
\end{verbatim}
ou encore de manière équivalente :
\begin{verbatim}
sum(x[j, i] for (j, i) in Iterators.product(J, I))
\end{verbatim}

\subsubsection{Définition d'ensembles et indexation}

Spécifier les variables une à une est vite fastidieux, pour ne pas dire irréaliste (par exemple si nous avons un million de variables), c'est pourquoi en modélisation algébrique, nous utilisons des indices.
Julia étant un langage de programmation à part entière, il ne possède pas de commande spécifique \verb|SET|, mais utilise les types de données standards comme les vecteurs (\verb|Vector|) ou les ensembles (\verb|Set|) pour représenter les indices.
Par exemple, l'ensemble d'indices $\lbrace 1,2,\ldots,10 \rbrace$ se définit simplement par :
\begin{verbatim}
indices = 1:10
\end{verbatim}
Nous pouvons indexer nos variables JuMP par n'importe quel ensemble ou tableau de chaînes de caractères, nombres, symboles, etc.
Contrairement à GAMS, nous n'avons pas besoin d'un concept d'\verb|ALIAS| pour parcourir le même ensemble avec un autre indice. Nous pouvons simplement utiliser un nom de variable locale différent dans notre boucle ou notre expression de somme :
\begin{verbatim}
# Déclaration de variables indexées par deux ensembles I et J
@variable(model, x[i in I, j in J] >= 0)

# Traduction de la contrainte : x_ij + x_ji = 1 pour i, j dans 1..n
@constraint(model, [i in 1:n, j in 1:n], x[i, j] + x[j, i] == 1)
\end{verbatim}
Ici, Julia comprend naturellement que \verb|i| et \verb|j| parcourent le même ensemble d'indices \verb|1:n|.

\subsubsection{Entrée de données}

En Julia, les données sont stockées dans des structures de données standards du langage (variables, vecteurs, matrices, dictionnaires, etc.) :
\begin{description}
\item[Scalaires], pour les valeurs simples (par ex. \verb|valeur = 100|).
\item[Dictionnaires (\texttt{Dict})], très pratiques pour associer des valeurs à des indices nommés sous forme de vecteurs (par ex. \verb|Dict("A" => 10, "B" => 20)|).
\item[Matrices (\texttt{Matrix}) ou Tableaux multidimensionnels], pour des données indexées à deux dimensions ou plus (par ex. \verb|A = [1 2; 3 4]|).
\item[Dictionnaires à clés de type n-uplets (tuples)], qui permettent d'indexer des paramètres multidimensionnels par des indices nommés de manière similaire à une table (par ex. \verb|C = Dict(("A", 1) => 94, ("B", 2) => 10)|).
\end{description}

L'exemple~\ref{jump_simple} peut ainsi se reformuler de façon indexée (ou compacte) comme suit :
\begin{verbatim}
using JuMP
using HiGHS

# Ensembles d'indices
I = ["X1", "X2", "X3"]

# Paramètres
c = Dict("X1" => 109.0, "X2" => 90.0, "X3" => 115.0)
a = Dict("X1" => 6.0,   "X2" => 4.0,  "X3" => 8.0)

# Initialisation du modèle
model = Model(HiGHS.Optimizer)

# Déclaration des variables de décision indexées par l'ensemble I
@variable(model, x[i in I] >= 0)

# Déclaration de la fonction objectif
@objective(model, Max, sum(c[i] * x[i] for i in I))

# Déclaration des contraintes
@constraint(model, Equation2, sum(x[i] for i in I) <= 100)
@constraint(model, Equation3, sum(a[i] * x[i] for i in I) <= 500)

# Résolution
optimize!(model)
\end{verbatim}

\subsection{Autres logiciels}

Les solutions logicielles retenues sont loin d'être les seules envisageables, et les choix sont volontairement orientés afin de répondre aux objectifs d'un cours d'introduction, mais ne représentent pas forcément un choix optimal.
Il existe d'autres langages de modélisation intéressants (on pensera par exemple à AMPL ou Pyomo), d'autres solveurs commerciaux ou gratuits.
Nous nous désintéresserons cependant des solveurs intégrés dans les tableurs, les algorithmes proposés par défaut dans ceux-ci étant en général de faible qualité, en particulier en optimisation non-linéaire, et limités dans le nombre de variables qui peuvent être traitées.
Il est toutefois bon de noter que des outils existent pour importer/exporter des données entre un tableur et des outils de modélisation.
